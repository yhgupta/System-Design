CSR vs SSR

Client-Side Rendering vs Server-Side Rendering — when to use each, how they work, and why it matters.

// how client-side rendering works

🌐
1. Request
Browser requests the page from server
📄
2. Bare HTML
Server returns a near-empty HTML shell with a <div id="root">
📦
3. JS Bundle
Browser downloads and parses the JavaScript bundle
🔌
4. Fetch Data
JS calls APIs to fetch actual content and data
✨
5. Render
JS renders the full UI in the browser

Best for

  • Dashboards & admin panels
  • Apps behind a login screen
  • Highly interactive UIs (real-time, drag-drop)
  • Single-page apps (SPAs)
  • When SEO doesn't matter

Avoid when

  • You need good SEO
  • Users have slow devices or connections
  • Fast first-paint is critical
  • Social media link previews matter
  • Public-facing marketing pages

// how server-side rendering works

🌐
1. Request
Browser requests the page from server
⚙️
2. Server Work
Server fetches data & renders the full HTML page
📄
3. Full HTML
Complete, content-filled HTML is sent to browser
👁️
4. Visible
User sees content immediately — no blank screen
⚡
5. Hydrate
JS loads and "hydrates" — attaches interactivity

Best for

  • Marketing & landing pages
  • E-commerce product pages
  • Blogs, news, content sites
  • Pages that need good SEO
  • Public-facing content

Avoid when

  • Page is highly interactive / stateful
  • Real-time updates are needed
  • Server resources are limited
  • Content is behind auth (waste of server render)
  • Complex app-like behavior needed

Client-Side Rendering

The browser does the heavy lifting. Server sends a blank shell + JS bundle. The app lives and breathes in the browser.

Server-Side Rendering

The server does the heavy lifting. Every page request triggers a full render server-side. Browser gets ready-to-display HTML.
